### Patches applied

1. **`rslib/i18n/src/lib.rs`** — Added `#![allow(text_direction_codepoint_in_literal)]` at crate root. Rust 1.96 is stricter about bidirectional Unicode codepoints in string literals (RTL Persian/Arabic translations). Upstream anki has this in `generated.rs` but it needs to be on `lib.rs` for newer compilers. Will need to send upstream PR.

### Build system notes

- The anki monorepo uses protobuf (`prost-build`) to generate `backend.rs` from `.proto` files. The generated file (6393 lines) provides a method-dispatch RPC interface: each method takes protobuf-encoded bytes and returns protobuf-encoded bytes. This is perfect for a NAPI bridge.
- Git submodules required: `ftl/core-repo` (i18n translations), `ftl/qt-repo` (desktop translations). Both needed for the build script to find FTL files.
- `out/` directory must contain `buildhash` and `rslib/proto/descriptors.bin` — these are pre-generated by the anki build system. In our vendored approach, we'll need to generate these once and check them in, or run protoc as part of our build.

## Architecture Decision: Vendoring Strategy

Two options for using rslib in our project:

**Option A: Git submodule** (recommended)
- Add anki as a submodule at `rust/vendor/anki`
- Use path dependency in our `rust/bridge/Cargo.toml`
- Pros: Easy to track upstream updates, can cherry-pick fixes
- Cons: Full repo history (mitigated by shallow clone)

**Option B: Extract rslib + deps only**
- Copy only `rslib/`, `ftl/`, `proto/`, relevant `Cargo.toml` snippets
- Pros: Minimal repo size
- Cons: Harder to merge upstream changes, may miss build infrastructure

Recommendation: **Option A** — submodule with `[patch]` section to point to local checkout.
